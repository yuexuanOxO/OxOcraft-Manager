# 文件撰寫規範（Documentation Guidelines）

> 定義 OxOcraft-Manager 文件的撰寫方向，確保所有文件保持一致的品質、風格與閱讀體驗。

---

# 文件資訊

| 項目   | 內容         |
| ---- | ---------- |
| 文件版本 | v1.0.0     |
| 狀態   | 正式啟用       |
| 適用範圍 | 所有開發文件     |
| 最後更新 | YYYY/MM/DD |

---

# 1. 文件的目的

文件的存在不是取代程式碼。

而是回答程式碼無法回答的問題。

例如：

* 為什麼這樣設計？
* 功能解決什麼問題？
* 模組之間如何合作？
* 為什麼選擇這個方案？

程式碼負責描述「怎麼做」。

文件負責描述「為什麼這樣做」。

---

# 2. 一份文件只回答一個主題

每份文件都應有明確定位。

例如：

* project-structure.md：介紹專案結構。
* backend.md：介紹 Backend 架構。
* frontend.md：介紹 Frontend 架構。
* whitelist.md：介紹白名單功能。

避免一份文件同時介紹多個不相關主題。

---

# 3. 一個章節只回答一個問題

每個章節都應聚焦於一個重點。

例如：

* 為什麼使用 Database？
* 為什麼選擇 SQLite？
* Database 的角色。

避免同一章節同時回答多個問題，降低閱讀負擔。

---

# 4. 先說「為什麼」，再說「怎麼做」

介紹功能或設計時，優先說明：

* 為什麼需要？
* 為什麼這樣設計？

之後再介紹：

* 如何運作？
* 如何實作？

讓讀者先理解設計目的，再理解實作方式。

---

# 5. README 與 Wiki 的定位不同

README 是專案首頁。

目的在於：

* 快速了解專案
* 展示主要功能
* 建立第一印象

Wiki（開發文件）則負責：

* 系統架構
* 設計理念
* 功能流程
* 技術細節

README 不應取代 Wiki。

---

# 6. 只介紹已完成的功能

README 僅展示目前已完成且可使用的功能。

尚未完成、名稱尚未確定或仍在規劃中的功能，不應提前加入 README。

避免 README 與實際產品內容不一致。

---

# 7. 圖片優先於文字

功能介紹時，圖片應作為主要展示方式。

文字只負責補充圖片無法表達的資訊，而不是重複描述畫面內容。

每張圖片都應回答一個問題，例如：

* 首頁長什麼樣？
* 白名單如何管理？
* 備份介面如何操作？

---

# 8. 不記錄短期資訊

避免在文件中記錄容易變動的資訊，例如：

* 固定數值
* 暫時性的流程
* 尚未定案的設計

文件應優先描述長期有效的設計理念。

---

# 9. 文件應與專案一起成長

文件不需要等到功能全部完成才開始撰寫。

建議開發流程：

```text
完成功能
    ↓
整理程式
    ↓
更新文件
```

讓文件持續反映專案目前的狀態，而不是一次性完成。

---

# 10. 保持簡潔

文件應優先追求：

* 容易閱讀
* 容易理解
* 容易維護

避免過度冗長或重複描述相同內容。

如果一段內容同時回答多個問題，應考慮拆成不同章節。

---

# 11. 最終原則

> **文件不是為了證明專案有多複雜，而是讓未來的自己與其他開發者，更容易理解這個專案。**

每一份文件，都應該讓閱讀者比閱讀前更理解 OxOcraft-Manager，而不是增加更多疑問。
