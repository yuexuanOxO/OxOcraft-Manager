# 版本管理規範（Version Policy）

> 定義 OxOcraft-Manager 的版本編號方式與版本提升原則，讓每次 Release 都有一致的標準。

---

# 文件資訊

| 項目   | 內容               |
| ---- | ---------------- |
| 文件版本 | v1.0.0           |
| 狀態   | 正式啟用             |
| 適用範圍 | OxOcraft-Manager |
| 最後更新 | YYYY/MM/DD       |

---

# 1. 目的

版本號除了表示更新順序，更代表專案目前的完成度與穩定性。

本文件用來定義 OxOcraft-Manager 的版本提升原則，避免不同版本之間缺乏一致的命名方式。

---

# 2. 版本格式

OxOcraft-Manager 採用三段式版本號：

```text
vMAJOR.MINOR.PATCH
```

例如：

```text
v0.9.0
v0.9.1
v0.10.0
v1.0.0
```

---

# 3. MAJOR（主版本）

主版本代表專案的重要里程碑。

通常發生於：

* 正式版發布（v1.0.0）
* 系統架構重大調整
* 發生不相容的重大變更

主版本更新頻率最低。

---

# 4. MINOR（次版本）

次版本代表功能上的成長。

例如：

* 新增主要功能
* 新增完整模組
* 大型重構
* UI / UX 明顯改善

只要讓專案能力有明顯提升，即可提升 Minor。

---

# 5. PATCH（修正版）

Patch 主要用於日常維護。

例如：

* Bug 修正
* 小型 UI 調整
* 效能改善
* 文件更新
* 小幅功能優化

Patch 不應包含大型功能變更。

---

# 6. 版本提升原則

| 更新內容     | 建議版本  |
| -------- | ----- |
| Bug 修正   | PATCH |
| 小型 UI 調整 | PATCH |
| 文件更新     | PATCH |
| 新增功能     | MINOR |
| 大型重構     | MINOR |
| 架構重大調整   | MAJOR |
| 正式版發布    | MAJOR |

若一次更新包含多種類型，應以影響最大的項目作為版本提升依據。

---

# 7. v0.x 與 v1.x 的差異

## v0.x

代表專案仍持續開發中。

功能、介面與架構皆可能持續調整。

重點放在：

* 功能建立
* 功能完善
* 使用體驗改善

---

## v1.x

代表專案進入正式版本。

主要功能已趨於穩定。

後續更新以：

* 新功能
* 功能優化
* Bug 修正

為主，避免頻繁修改核心架構。

---

# 8. 發布原則

每次 Release 前，應至少完成以下項目：

* 功能測試
* Bug 修正
* README 更新（如有需要）
* 相關文件更新（如有需要）
* Changelog 更新（如有需要）

---

# 9. 最終原則

> **版本號代表的是專案的成熟度，而不是更新次數。**

只有當專案真正產生有意義的改變時，才應提升版本，而不是為了發布新版本而增加版本號。
