## 1. 目的

本文件用來記錄 OxOcraft-Manager 的程式撰寫方向。

目的不是建立複雜的企業級架構，而是讓專案在單人維護的情境下，保持：

* 容易理解
* 容易修改
* 容易除錯
* 不過度抽象
* 功能邏輯集中

---

## 2. 基本原則

OxOcraft-Manager 採用「功能導向」的開發方式。

也就是：

> **以功能為單位整理程式，而不是為了架構而拆分程式。**

例如：

```text
player_ban/
player_permissions/
server_settings/
backup_service.py
```

每個模組應盡可能對應一個明確功能。

---

## 3. 不為了架構而架構

本專案不追求過度複雜的分層設計。

如果一段程式目前只有單一功能會使用，且放在原本模組內仍然清楚，就不需要急著抽離。

只有在以下情況，才考慮抽成共用函式或共用模組：

* 多個功能都需要使用
* 重複邏輯已經造成維護困難
* 同一段流程修改時需要同步修改多處
* 抽離後能明顯提升可讀性

例如：

多個功能都需要讀取 Minecraft JSON 檔案時，才適合抽成共用 JSON 讀取工具。

---

## 4. Route 與 Service 的分工

Route 負責接收請求，Service 負責完成事情。

### Route 主要負責

* 接收 Request
* 取得參數
* 基本驗證
* 呼叫對應 Service
* 回傳 Response

Route 不應放入大量功能流程。

### Service 主要負責

* 判斷狀態
* 執行功能流程
* 呼叫 RCON
* 讀寫 JSON
* 更新 Database
* 寫入歷史紀錄
* 回傳處理結果

這樣可以讓相同功能不只被 API 呼叫，也能被排程、同步流程或其他 Backend 模組重複使用。

---

## 5. 功能模組應保持完整

功能相關的流程應盡量集中在同一個模組內。

例如：

Whitelist 相關邏輯應集中於 Whitelist Service。

Player Ban 相關邏輯應集中於 Player Ban Service。

Backup 相關邏輯應集中於 Backup Service。

這樣未來需要修改功能時，可以快速找到相關程式，而不需要在大量零散檔案中搜尋。

---

## 6. 共用功能才抽離

若功能只被單一模組使用，不需要過早抽成共用模組。

適合抽離的例子：

* JSON 讀寫
* 時間格式化
* Database 連線
* RCON 呼叫
* 玩家 UUID 判斷
* 共用 Response 格式

不適合過早抽離的例子：

* 只被白名單使用的新增流程
* 只被黑名單使用的封鎖判斷
* 只被備份功能使用的備份細節

---

## 7. Frontend 模組原則

前端 JavaScript 應以功能為單位拆分。

例如：

```text
player_ban.js
player_whitelist.js
player_permissions.js
server_settings.js
backup.js
```

每個檔案應主要負責自己的功能畫面與互動。

若多個功能都需要相同能力，再考慮抽成共用模組。

例如：

* 玩家頭像
* 系統 Dialog
* Notification
* Server UI State
* Offline Default Skins

---

## 8. CSS 結構原則

CSS 應依照用途分類。

目前主要分為：

```text
base/
components/
layout/
pages/
```

原則如下：

* `base/`：全域基礎樣式
* `components/`：共用元件樣式
* `layout/`：主要排版
* `pages/`：單一功能頁樣式

功能頁專用樣式應優先放在 `pages/`，避免影響其他功能。

---

## 9. 命名原則

命名應以清楚為優先。

不需要過度縮寫。

建議：

* 檔名能看出功能
* 函式名稱能看出用途
* 變數名稱能看出資料意義

避免使用：

* 過短縮寫
* 不明確名稱
* 與 Minecraft 原生命名衝突的名稱

---

## 10. 重構原則

重構應以降低維護成本為目標。

不應為了讓架構看起來更專業而重構。

適合重構的情況：

* 重複邏輯太多
* 功能流程難以理解
* Bug 容易重複發生
* 修改一處需要同步修改多處
* 模組責任混亂

不適合重構的情況：

* 只是想讓檔案數量變多
* 只是想套用大型專案架構
* 原本流程已經清楚且穩定

---

## 11. 開發節奏

OxOcraft-Manager 採用以下開發節奏：

```text
先完成
    ↓
測試
    ↓
修正問題
    ↓
必要時重構
    ↓
更新文件
```

不要求一開始就設計成最終架構。

但當功能逐漸穩定後，應逐步整理程式碼，避免技術債累積過多。

---

## 12. 最終原則

> **程式應該讓未來的自己容易理解，而不是只讓當下的功能能跑。**

OxOcraft-Manager 的程式碼應保持簡單、直接、可維護。

不追求過度抽象，也不追求大型專案架構。

只在真正需要時重構，只在真正共用時抽離。
